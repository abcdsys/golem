# Reread 插件

## 🔄 功能简介

Reread 插件实现了"人类的本质是复读机"的功能，能够智能检测群聊中的重复消息并自动复读。当同一条消息连续出现两次时，插件会自动发送第三次，形成经典的"复读机"效果。

## ✨ 主要特性

- **智能复读**：自动检测连续重复的消息
- **防止重复**：同一条消息只复读一次，避免无限循环
- **多类型支持**：支持文本消息和表情消息
- **群聊感知**：按发送者分组，独立追踪每个群聊的消息状态

## 🚀 工作原理

### 复读逻辑

```
用户A: 早上好           -> 缓存消息
用户B: 早上好           -> 检测到重复，触发复读
机器人: 早上好          -> 复读完成，标记已复读
用户C: 早上好           -> 已复读过，不再响应
```

### 状态管理

插件维护两个状态映射表：

1. **cache（消息缓存）**：记录每个发送者的最新一条消息
2. **read（已复读记录）**：记录已经复读过的消息，防止重复复读

### 触发条件

复读触发需要同时满足以下条件：

1. ✅ 当前消息与缓存中的上一条消息**完全相同**（内容和类型都匹配）
2. ✅ 该消息**尚未被复读过**
3. ✅ 消息类型为**文本**或**表情**

## 🔧 技术细节

### 消息匹配机制

插件使用 `reflect.DeepEqual` 进行消息比对，确保：
- 消息类型完全一致（文本 vs 表情）
- 消息内容完全一致（逐字符比较）

### 状态隔离

每个发送者（群聊/私聊）的消息状态独立管理：
```go
type data struct {
    typ     string  // 消息类型
    content string  // 消息内容
}

read  map[string]data  // 按发送者ID索引的已复读记录
cache map[string]data  // 按发送者ID索引的消息缓存
```

### 执行流程

```
接收消息
   ↓
检查是否已复读过？
   ↓ 是 → 跳过（避免重复）
   ↓ 否
检查是否与缓存消息相同？
   ↓ 是 → 发送复读消息 → 标记已复读
   ↓ 否
更新消息缓存 → 等待下次匹配
```

## ⚠️ 注意事项

1. **单次复读**：同一条消息只会被复读一次，避免陷入无限循环
2. **精确匹配**：只有完全相同的消息才会触发复读（包括空格、标点等）
3. **类型区分**：文本消息和表情消息分开处理，不会交叉匹配
4. **内存管理**：
   - 插件会在内存中维护消息缓存和已复读记录
   - 长时间运行可能会积累较多状态数据
   - 重启插件会清空所有状态
5. **群聊隔离**：不同群聊/私聊的复读状态相互独立

## 🔍 故障排查

### 常见问题

1. **没有触发复读**
   - 检查消息是否**完全相同**（包括空格、大小写等）
   - 确认该消息是否已经被复读过
   - 检查消息类型是否为文本或表情

2. **复读了不该复读的消息**
   - 检查是否有多个用户连续发送了相同消息
   - 确认插件状态是否正常

3. **复读后又复读了一次**
   - 通常不会发生，因为有已复读记录保护
   - 如果出现，可能是插件重启导致状态清空

### 调试日志

插件会输出以下调试日志：

```

```

可以通过日志查看插件的运行状态。
